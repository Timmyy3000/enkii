Enkii · third review lane

Repository-defined policy review

A repository-owned prompt tells Enkii how to inspect the PR, which engineering guide to read, and how the team wants findings reported. It runs automatically beside code and security review.

Product boundary

Prompt, not policy storage

The configured file is the policy agent's complete instruction layer. It can say which guide to read, which risk profile applies, what counts as a finding, and how citations should look.

Team ownership

Guide stays discoverable

The engineering guide remains wherever engineers already expect it. Enkii does not ingest it through a special action input or force a universal rule-ID format.

Runtime topology

One PR event, three independent lanes

The policy prompt is loaded after PR HEAD is checked out. Its referenced guide is not preloaded; the agent reads it when following the prompt.

Scope control

What v1 includes

In scope

  • Optional policy prompt path; path presence enables the lane
  • Optional policy model inheriting the code-review model
  • HEAD-sourced prompt for same-repository PRs
  • Automatic concurrent execution and independent posting
  • Shared candidate/validator contracts
  • Policy-specific artifacts, marker, label, and output ID
  • Safe path validation and explicit fork behavior

Out of scope

  • No /policy or /style command
  • No cross-lane deduplication or arbitration
  • No hard-coded rule-ID or citation schema
  • No direct style-guide action input
  • No mechanical merge blocking
  • No universal policy methodology owned by Enkii

Acceptance focus

Observable behavior

  • Empty prompt path means no policy task and no policy review.
  • Configured same-repo PRs use the prompt present at PR HEAD.
  • The agent can read any repository file allowed by existing read-only tools.
  • Prompt-defined citations survive unchanged in candidate bodies.
  • Prompt-defined citations and semantic wording survive the final GitHub formatting envelope.
  • Complete finding bodies survive spillover, unanchored, and summary-only retry paths.
  • Policy reviews omit Enkii's generic mergeability score and verdict.
  • Policy output uses its own marker and policy_review_id.
  • Code and security results are unchanged when policy is absent.
  • Successful lanes still post when another lane fails during loading, execution, validation, or GitHub posting.

Risk register

Known tradeoffs, made explicit

RiskImpactMitigation / stance
Self-amending policyA PR can change the prompt or guide used to review itself.Intentional HEAD behavior. Make changes visible in the diff and recommend CODEOWNERS for policy files.
Fork-controlled promptUntrusted contributors could redefine agent instructions.Recommended v1: skip policy only, continue other lanes, leave the policy output empty, and explain the skip in the tracking comment.
Flexible outputEnkii cannot mechanically guarantee a team's citation convention.Prompt owns the convention; shared schema still constrains paths, lines, severities, and bodies.
Cost and latencyA third agent, and optionally validator, adds spend and may become the slowest lane.Opt-in by non-empty path; run concurrently; document model inheritance and validator cost.
Hard-coded two-lane UIPolicy could inherit incorrect “code review” labels or merge language.Generalize review-kind presentation and add regression tests for all three labels.
One lane failsCurrent Promise.all behavior can prevent every completed review from posting.Settle lanes independently, post successes, then fail the action with lane-specific diagnostics.
One GitHub post failsCurrent sequential posting aborts before later lanes post.Independently settle every post, retain successful output IDs, then report aggregated lane errors.
Fallback truncates policyCurrent spillover/unanchored summaries keep only the first line.Policy-specific fallback rendering preserves full finding bodies and citations; omit generic mergeability scoring.
Path escape or symlinkString-prefix containment and stat() are insufficient.Use a boundary-safe relative check plus symlink-aware inspection; cover sibling-prefix and symlink cases.

Resolved decision

Fork PR behavior

Skip only the policy lane when the configured prompt would come from fork-owned HEAD.

Code and security continue, policy_review_id remains empty, the tracking comment explains the skip, and an otherwise successful run does not fail. There is no bundled policy fallback.

Resolved decision

Public action contract

policy_review_skill_path accurately communicates that the configured Markdown is the agent's prompt/skill, not the engineering guide.

The companion policy_review_model inherits review_model when empty. No separate boolean is needed because path presence is the opt-in switch.
